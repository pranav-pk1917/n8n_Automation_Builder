# Audit: WF-01-Ingest

## 1. Current Logic Assessment
This workflow ingests SEMrush CSV keyword dumps via a webhook (`/webhook/wf-01-ingest`). It validates the payload, decodes optional base64 content, parses CSV into keyword records, checks against existing keywords and a "graveyard" list for deduplication, chunks records into batches of 500, and bulk-inserts into the `raw_keywords` table. Overall health is **fair** — the core logic works for small-to-medium datasets but has hard limits that silently truncate data above 50K keywords and a fragile CSV parser that breaks on edge cases.

## 2. Identified Shortcomings & Doubts
- [ ] Issue 1: **Hard Range Cap at 50K Keywords (CRITICAL for Scale):** The `Fetch_Existing_Keywords` Supabase call uses `Range: 0-49999`. For a client with 60K+ existing keywords, any keywords beyond position 50,000 are invisible to the deduplication logic, causing duplicate inserts that fail or create noise. There is no pagination loop to handle the full keyword set.
- [ ] Issue 2: **Graveyard Dedupe Is Dead Code:** The `Fetch_Graveyard_IDs` node fetches graveyard keyword IDs into a set via `Build_Graveyard_Set`, but the `Dedupe_Keywords` node only dedupes against the `existingSet` (from `Fetch_Existing_Keywords`). The `graveyard_ids` set is built but never actually referenced in the filter logic. Keywords in the graveyard will still be inserted as new.
- [ ] Issue 3: **Fragile Manual CSV Parser:** The `Parse_CSV` node implements a custom CSV parser that handles quoted fields but has known failure modes: fields containing escaped quotes (`""`), multi-line quoted fields, and fields with embedded delimiters other than commas (SEMrush sometimes uses semicolons or tabs). A single malformed line crashes the entire ingest.
- [ ] Issue 4: **Base64 Detection Heuristic Is Unreliable:** The validation code checks `if (!csvText.includes(',') && !csvText.includes('\n'))` to detect base64. This produces a false positive for a CSV with a single keyword row (no newlines, no commas — e.g., a one-column export). The decryption will produce garbage.
- [ ] Issue 5: **Pipeline Run Completes Prematurely:** The `Complete_Ingest_Run` node runs once per chunk (both from the IF true and false branches). The first chunk to reach it writes `status: 'completed'`. If the second chunk fails during `Bulk_Insert_Keywords`, the pipeline run is already marked completed, and the failure is invisible.
- [ ] Issue 6: **No Insert Error Handling or Retry:** If `Bulk_Insert_Keywords` fails (duplicate key violation, Supabase rate limit, or timeout), the workflow has no retry mechanism. The failed chunk's records are lost, but `Complete_Ingest_Run` still reports success.

## 3. Scope for Improvement
Fixing the graveyard dedupe is a one-line code edit. The 50K range cap needs a pagination loop (or a Supabase function that does the dedupe server-side). The CSV parser should be replaced with a proper library or, at minimum, handle the most common edge cases. Adding a final "all chunks done" aggregation step before marking the run as complete would make the ingest atomic.

## 4. Execution Plan (For Next Agent)
1. **Fix Graveyard Deduplication:** In the `Dedupe_Keywords` Code node, add the graveyard check. After building `existingSet`, also filter on `graveyard_ids`. Since the graveyard IDs are `raw_keyword_id`s (not keyword text), you need to fetch the actual keywords for those IDs, or change the graveyard table to store keyword text. Quick fix: add a second fetch after `Fetch_Existing_Keywords` that resolves graveyard IDs to keyword text, then dedupe against both. Better fix: create a Supabase RPC function `check_keyword_exists(p_client_id, p_keywords[])` that does both checks server-side.
2. **Add Pagination for Large Keyword Sets:** Replace the single `Fetch_Existing_Keywords` GET call with a loop pattern: first GET returns up to 50K, check the `content-range` header. If there are more results (total > 50K), use SplitInBatches with a Code node that fetches subsequent pages with `Range: 50000-99999`, `Range: 100000-149999`, etc. Alternatively, move deduplication to a Supabase RPC that handles the full table scan server-side.
3. **Harden the CSV Parser:** In `Parse_CSV`, replace the manual parser with a two-step approach: first try a simple split (fast path for clean CSVs), and on failure, use a more robust parser. At minimum, handle double-quote escaping (`""` → `"`) and differentiate between comma and tab/semicolon delimiters by analyzing the header row character distribution.
4. **Make Base64 Detection Explicit:** Instead of guessing, require a `csv_encoding` field in the webhook payload: `{ client_id, csv_content, csv_encoding: "base64" | "plain" }`. Default to `"plain"` if not provided, removing the heuristic entirely.
5. **Make Run Completion Atomic:** Move `Complete_Ingest_Run` to execute only once after ALL chunks finish. Use a SplitInBatches node (or an Aggregate node after the chunk loop) to collect all chunk results, then write `status: 'completed'` with the full stats. The current flow chain should be: `Chunk_Records` → `SplitInBatches` → `IF_Has_Records` → `Bulk_Insert_Keywords` → loop back → on done → Aggregate chunk results → `Complete_Ingest_Run`.
6. **Add Error Accumulation:** In the Code node after `Bulk_Insert_Keywords`, check the Supabase response for errors. Accumulate failed records into an error summary. If any chunk fails, mark the pipeline run as `status: 'partial'` with `output_summary.errors` detailing the failed records.
