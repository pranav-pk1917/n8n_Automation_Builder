# Audit: WF-05-Sheet-Sync

## 1. Current Logic Assessment
This workflow runs nightly at 03:00 UTC to sync scored clusters, taxonomy suggestions, and API cost data from Supabase to a Google Sheet per client. It fans out per active client, fetches clusters/suggestions/costs in parallel, builds per-tab data structures, and writes to Google Sheets. A Slack notification is sent per client. Overall health is **fair but incomplete** — the workflow is partially built: only the "Cost" tab is written to Sheets, despite the sticky note promising 7 tabs. The notification logic is noisy for unconfigured clients.

## 2. Identified Shortcomings & Doubts
- [ ] Issue 1: **Only Cost Tab Is Actually Written:** The `Build_Sheet_Data` Code node builds `pillar_tabs`, `themes_rows`, `tax_rows`, and `cost_rows`. However, only ONE Google Sheets node (`Write_Cost_Tab`) exists, and it only writes `cost_rows` to the "Cost" tab. The per-pillar tabs, themes tab, taxonomy-suggestions tab, audit tab, and cross-validation tab described in the sticky note are data that gets built but never written. The workflow claims to deliver 7 tabs but delivers 1.
- [ ] Issue 2: **Slack Notification for Unconfigured Clients:** The `Send_Sync_Notification` node fires for EVERY client, including those without a `sheets_view_id`. The IF_Has_Sheets_ID node correctly routes clients with a sheets_id to `Write_Cost_Tab`, but the false branch also goes directly to `Send_Sync_Notification`. For unconfigured clients, this posts a Slack message saying "Sheet sync complete" with no actual data synced.
- [ ] Issue 3: **No Range Header on Clusters Fetch:** `Fetch_Client_Clusters` GET has no Range header. If a client has >1000 clusters (Supabase default page size), clusters beyond page 1 are omitted from the sync.
- [ ] Issue 4: **Parallel Fetches Have No Error Isolation:** `Fetch_Client_Clusters`, `Fetch_Taxonomy_Suggestions`, and `Fetch_API_Costs` run in parallel from `Split_Clients`. If any one of these three fetch calls fails (timeout, auth error, empty response), the entire `Build_Sheet_Data` node may fail because it expects all three inputs. There's no fallback for partial data.
- [ ] Issue 5: **Write_Cost_Tab Uses `appendOrUpdate` Without Clear Logic:** The `Write_Cost_Tab` node uses `operation: "appendOrUpdate"` with `mappingMode: "autoMapInputData"`. The "Cost" tab gets appended to every night, but there's no deduplication or clearing of old data. After 30 days, the Cost tab has 30 duplicate rows per client. The `appendOrUpdate` operation's update behavior depends on having a unique key column — without one, it's pure append.
- [ ] Issue 6: **All Pillar Tabs Data Is Computed but Unreachable:** The `pillar_tabs` array in Build_Sheet_Data contains cluster rows organized by pillar — exactly the data that should populate the per-pillar tabs. But since Write_Cost_Tab only writes `cost_rows`, this computed data is completely unused. The Code node spends compute time building data that's never written.
- [ ] Issue 7: **Hardcoded Slack Channel:** The `Send_Sync_Notification` hardcodes `channel: 'C0B43A7QG5P'`. If the Slack channel ID changes or if different clients need different channels, this requires a code edit.

## 3. Scope for Improvement
This workflow needs the remaining 6 tabs implemented. Each tab should be a Google Sheets node (or multiple nodes if the Sheets node only writes one tab at a time). The notification logic needs to distinguish between "synced data" and "nothing to sync." Error isolation on parallel fetches prevents one failed query from blocking the other tabs.

## 4. Execution Plan (For Next Agent)
1. **Add Per-Pillar Tab Writing:** After `Write_Cost_Tab`, add Google Sheets nodes for each pillar tab. Since pillars vary per client, use a Code node to iterate `pillar_tabs` and build writes per tab. Or, add a SplitInBatches node that iterates over the `pillar_tabs` array, with one Google Sheets node inside the loop that writes to the dynamic tab name `{{ $json.tabName }}`. The first write to each tab should use `operation: "create"` (to create the tab if missing) and subsequent writes use `operation: "append"`.
2. **Add Themes Tab, Taxonomy-Suggestions Tab, and Cost Tab (completing 4+ tabs):** The `themes_rows` and `tax_rows` also need Google Sheets write nodes. For the "Themes" tab, write `[[theme, count], ...]`. For "Taxonomy-Suggestions", write `[[kind, suggested_value, rationale, created_at], ...]`. Add these as additional parallel branches after `Build_Sheet_Data`.
3. **Fix Slack Notification for Unconfigured Clients:** In `IF_Has_Sheets_ID`, change the false branch to NOT send a Slack notification. Instead, add a Note/Warning that silently skips the client. Optionally, accumulate unconfigured client names and send one summary Slack message at the end: "Sheet sync complete for N clients. M clients skipped (no sheets_view_id configured)."
4. **Add Range Header and Pagination:** On `Fetch_Client_Clusters`, add `Range: 0-49999` to handle up to 50K clusters. Same for `Fetch_Taxonomy_Suggestions` and `Fetch_API_Costs`. If pagination is needed, follow the WF-01 pattern.
5. **Add Error Isolation on Parallel Fetches:** Wrap each fetch (Fetch_Client_Clusters, Fetch_Taxonomy_Suggestions, Fetch_API_Costs) with `Continue on Fail` enabled. In `Build_Sheet_Data`, check if each input exists before using it: `const clusters = $('Fetch_Client_Clusters').first().json || [];`. Build tabs only for available data.
6. **Implement Sheet Deduplication:** For the Cost tab, change from `appendOrUpdate` to a clear strategy: (a) clear the tab with a separate "clear sheet" operation (or use `operation: "overwrite"` with `writeMode: "replace"`), then (b) append fresh data. Alternatively, use a "Sync Time" column as a batch identifier and filter out old rows in Supabase before building the sheet data.
7. **Make Slack Channel Configurable:** Replace the hardcoded channel ID with `{{ $json.client_config?.hitl_channels?.[0] || $env.SLACK_HITL_CHANNEL || 'C0B43A7QG5P' }}`.
