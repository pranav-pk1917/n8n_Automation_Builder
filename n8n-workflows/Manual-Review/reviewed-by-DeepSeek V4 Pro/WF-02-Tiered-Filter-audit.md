# Audit: WF-02-Tiered-Filter

## 1. Current Logic Assessment
This workflow runs a 3-tier keyword filter pipeline: Tier 0a (regex negatives via Supabase RPC), Tier 0b (competitor/navigational filter via Supabase RPC), Tier 1 (embedding similarity scoring vs gold examples), and Tier 2 (LLM classification of borderline keywords via Gemini). It posts Slack HITL cards for `needs_review` decisions. Overall health is **poor** — the workflow has a critical data-loss bug where Tier 1 passed/rejected keywords never get written to `keyword_classifications`, and the embedding batching logic has structural issues.

## 2. Identified Shortcomings & Doubts
- [ ] Issue 1: **Tier 1 Passed/Rejected Keywords Are Silently Dropped (CRITICAL):** After `Score_Tier1` computes decisions for all unclassified keywords, the `Prep_Tier2_Items` node only expands `borderline` keywords into individual items. The `passed_tier1` and `rejected_tier1` arrays are stored in the context but **never written to `keyword_classifications`**. Only Tier 2 borderline → LLM classified keywords reach `Write_Classification`. This means the majority of keywords (typically 60-80% that pass/reject cleanly at Tier 1) are processed by embeddings but never stored.
- [ ] Issue 2: **Tier 0 RPCs Execute in Parallel with Data Fetch (Race Condition):** `Merge_Client_Config` fans out to three parallel branches: `Tier0a_Regex_Filter`, `Fetch_Unclassified_Keywords`, and `Fetch_Gold_Examples`. Tier 0a runs and writes to the database while the fetch for unclassified keywords is also running. There is no guarantee Tier 0a completes before the unclassified fetch reads — keywords could be classified by Tier 0a AFTER they're already fetched as "unclassified", or vice versa. The Tier 0 RPCs need to complete before any downstream reads.
- [ ] Issue 3: **Embedding Batching Misalignment:** `Prepare_Tier1` stores the FULL `keywords_to_embed` array in a single item. `Batch_For_Embedding` (SplitInBatches, batchSize=100) then creates items with batch indices. The `Embed_Keywords_Batch` node uses `$json.keywords_to_embed.slice($json.batchIndex * 100, ...)` to select the right slice — but this assumes the context has the full array. The `batchIndex` is an integer from SplitInBatches (0, 1, 2...), and the slice looks correct. However, if any individual batch fails during embedding, the worker has no error recovery for that batch — the execution continues with the next batch but with corrupted data.
- [ ] Issue 4: **Hard 50K Range Cap on Unclassified Fetch:** `Fetch_Unclassified_Keywords` uses `Range: 0-49999`, same 50K hard limit pattern as WF-01. For clients with 60K+ raw keywords, up to 10K+ keywords are silently excluded from classification.
- [ ] Issue 5: **No Range Header on Gold Examples Fetch:** `Fetch_Gold_Examples` has no Range header. Supabase's default page size is 1000 rows. If a client accumulates >1000 gold examples, only the first 1000 are used for centroid computation, biasing the similarity scoring.
- [ ] Issue 6: **Cosine Similarity Duplicated Across Nodes:** The `cosine` function is defined identically in both `Prepare_Tier1` (line ~159) and `Score_Tier1` (line ~224). If the formula changes (e.g., adding a bias term), it must be updated in two places.
- [ ] Issue 7: **No LLM Rate Limiting for Tier 2:** The `LLM_Tier2_Classify` node fires one OpenRouter request per borderline keyword sequentially. With 200 borderline keywords, this sends 200 sequential API calls with no rate limiting, potentially hitting OpenRouter's rate limits or inflating costs unnecessarily.

## 3. Scope for Improvement
The critical fix is adding classification writes for Tier 1 results — this alone unblocks 60-80% of keyword throughput. The Tier 0 race condition needs reordering (sequentialize Tier 0 before fetches). The duplicate cosine function should be extracted to a shared helper or moved to a single node. For borderline keywords at Tier 2, batching multiple keywords into a single LLM call (instead of one per keyword) would reduce API calls by 10-20x and cut costs proportionally.

## 4. Execution Plan (For Next Agent)
1. **Write Tier 1 Passed/Rejected to keyword_classifications:** After `Score_Tier1`, add a new Code node `Write_Tier1_Classifications` that iterates `passed_tier1` and `rejected_tier1` arrays and creates classification records. Then add a Supabase bulk insert node (similar to `Write_Classification` but for Tier 1 results). The Tier 1 results have `tier1_decision`, `raw_keyword_id`, `pos_score`, `neg_score`, and `embedding` — all these fields should be written. Set `decided_by: 'tier1_embedding'`. This should run in parallel to the borderline path (not instead of it).
2. **Sequentialize Tier 0 Before Data Fetch:** Restructure `Merge_Client_Config` output to go to `Tier0a_Regex_Filter` → `Tier0b_Competitor_Filter` → THEN fan out to `Fetch_Unclassified_Keywords` and `Fetch_Gold_Examples`. Or, add a `Wait` node between Tier 0b and the fetches to ensure Tier 0 mutations commit before reading. Better yet: make Tier 0 RPCs return the filtered set of keyword IDs and use those as a filter on the fetch, making the dependency explicit.
3. **Add Pagination for Unclassified Fetch:** Use the same pagination pattern recommended for WF-01. Read the `content-range` header from the Supabase response. If more pages exist, loop through them and accumulate all unclassified keywords before passing to Prepare_Tier1.
4. **Add Range Header to Gold Examples Fetch:** Add `Range: 0-99999` on the `Fetch_Gold_Examples` GET call, or add a pagination loop if gold examples exceed 100,000 (unlikely but defensive).
5. **Deduplicate the Cosine Function:** Delete the cosine definition from `Score_Tier1` and instead compute similarity in `Prepare_Tier1` where it's already defined. Or, move all similarity computation to a single Code node that receives embedding results and gold centroids, outputting fully scored items.
6. **Batch Tier 2 LLM Calls:** Instead of `Prep_Tier2_Items` expanding to one item per borderline keyword, batch 20-30 borderline keywords into a single LLM call. Modify the system prompt to accept an array of keywords and return an array of classifications. This reduces API calls from N to N/25 and dramatically improves throughput. Use JSON array response format: return `{ results: [{ keyword, status, ... }] }`.
