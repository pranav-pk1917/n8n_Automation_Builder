# Audit: WF-ONBOARD

## 1. Current Logic Assessment
*Briefly define what the workflow currently does and its overall health.*
This workflow handles new client onboarding by crawling the client's sitemap, using an LLM to extract service pillars and ICP data, and sending a Slack questionnaire for human verification. It then cross-validates the human answers against the LLM analysis and writes the final configuration to Supabase. The overall health is poor due to a broken batching loop and webhook timeout issues.

## 2. Identified Shortcomings & Doubts
*List any unclear logic, potential failure points, or scalability bottlenecks.*
- [ ] Issue 1: **Broken SplitInBatches Loop:** `SplitInBatches_Pages` splits the top 20 URLs into batches of 5. However, the chain ends at `Aggregate_Page_Data` and never loops back. Only the first 5 pages are crawled and analyzed.
- [ ] Issue 2: **Webhook Timeouts:** The workflow uses `Wait_For_Questionnaire` and `Wait_For_Flag_Ack` as standard Webhook nodes with `responseMode: "lastNode"`. Because Slack interactions are asynchronous and can take hours, the initial `wf-onboard-start` webhook will time out, causing the caller to receive a 504 error.
- [ ] Issue 3: **Improper Wait Nodes:** Standard Webhook nodes cannot suspend and resume a running workflow execution. They start *new* executions. The workflow loses its context when the Slack button is clicked.

## 3. Scope for Improvement
*Explain how the efficiency and scalability can be boosted without over-engineering.*
The workflow must respond to the initial webhook immediately. The batch loop must be fixed to crawl all 20 pages. The Slack interaction points must use n8n's "Wait" node (configured for Webhook) to properly suspend the execution and resume it with the existing context when the user submits the form.

## 4. Execution Plan (For Next Agent)
*Provide step-by-step, actionable instructions on exactly which nodes to change, add, or remove.*
1. **Respond Early:** Add a `Respond to Webhook` node immediately after `Create_Pipeline_Run` to close the initial HTTP request and prevent timeouts.
2. **Fix the Batch Loop:** Re-route the output of `Extract_Page_Content` back to the input of `SplitInBatches_Pages`. Connect the `done` output branch of `SplitInBatches_Pages` to `Aggregate_Page_Data`.
3. **Replace Webhook Nodes with Wait Nodes:** Replace `Wait_For_Questionnaire` and `Wait_For_Flag_Ack` with `Wait` nodes configured to "Wait for Webhook Call". This allows the workflow execution to suspend and resume correctly when the Slack payloads arrive.
4. **Update Slack Action URLs:** Ensure the Slack interactive buttons send their payloads to the specific resume URLs generated by the new Wait nodes.
