## What this workflow does (plain English)

Not every keyword from a CSV is worth pursuing. This workflow runs a **three-tier filter** to decide pass / reject / needs human review:

| Tier | How it works | Speed / cost |
|------|----------------|--------------|
| **Tier 0** | Database rules (regex, blocklists) via Supabase RPC | Fast, free |
| **Tier 1** | Embedding similarity vs client pages (Supabase RPC) | Fast, low cost |
| **Tier 2** | LLM classifies **borderline** keywords only | Slower, uses OpenRouter |

Results are saved in `keyword_classifications`. Keywords marked `needs_review` get a **Slack card** with Approve/Reject buttons for a human.

---

## Current features (overview)

| Feature | What it does |
|---------|----------------|
| **Tier 0 RPC** | `apply_tier0_filters` — auto-rejects obvious junk (brands, navigational, etc.). |
| **Fetch unclassified** | Loads up to 500 `raw_keywords` still in `status: raw`. |
| **Tier 1 RPC** | `tier1_embed_and_score` — scores relevance; splits passed / borderline / rejected. |
| **Tier 2 LLM** | Gemini classifies each **borderline** keyword (intent, niche, theme hints). |
| **Classification writes** | Inserts into `keyword_classifications` (tier 1 or 2). |
| **HITL Slack cards** | Posts review card when LLM says `needs_review`. |
| **Webhook response** | Returns `{ ok: true }` when the run finishes. |

## Node reference

| Node | Type | What it does |
|------|------|--------------|
| **Sticky Note** | Sticky | Canvas documentation. |
| **Filter_Webhook** | Webhook | Starts the filter run. |
| **Validate_Input** | Code | Ensures `client_id` present. |
| **Apply_Tier0_Filters** | HTTP Request | POST RPC `apply_tier0_filters`. |
| **Fetch_Unclassified_Keywords** | HTTP Request | GET raw keywords for client. |
| **Call_Tier1_Embed_Score** | HTTP Request | POST RPC `tier1_embed_and_score` with keyword array. |
| **Flatten_Tier1_Results** | Code | Expands RPC result into per-keyword items. |
| **IF_Borderline** | If | `tier1_decision === borderline` → Tier 2 path. |
| **LLM_Tier2_Classify** | HTTP Request | OpenRouter chat for one borderline keyword. |
| **Prepare_T2_Write** | Code | Parses LLM JSON into classification fields. |
| **IF_Needs_Review** | If | Checks `decision === needs_review` (both branches still write). |
| **Write_T2_Classification** | HTTP Request | POST `keyword_classifications` tier 2. |
| **Send_HITL_Slack_Card** | HTTP Request | Slack Approve/Reject buttons. |
| **Write_T1_Classification** | HTTP Request | POST tier 1 for passed/rejected from Tier 1. |
| **Respond_Filter_OK** | Respond to Webhook | `{ ok: true }`. |

## Known limitation (important for testers)

The live workflow processes items in a **linear** graph. For many borderline keywords, Tier 2 runs **one keyword per execution path** as n8n walks items — large batches may need multiple runs or a future loop refactor. Start testing with a **small CSV** (10–20 keywords).
